## 示例 1：AI 产品拆解

### 用户输入

```
写一篇拆解 Cursor 的文章，分析它为什么能火
```

### 执行流程

#### 第 1 步：判断内容类型

- **类型**：AI 产品拆解
- **核心角度**：从产品设计角度分析 Cursor 为什么能在竞争激烈的 AI 编程工具市场突围
- **目标读者**：AI 爱好者、产品经理同行、想了解 AI 编程工具的开发者

#### 第 2 步：搜索资料（3-4 轮）

**第 1 轮 - 产品基本信息**：
```
查询："Cursor 官网"
查询："Cursor AI code editor features"
```

**第 2 轮 - 用户反馈和评价**：
```
查询："Cursor 用户评价 知乎"
查询："Cursor vs Copilot 对比"
查询："为什么选择 Cursor"
```

**第 3 轮 - 商业数据**：
```
查询："Cursor 融资 用户数 ARR"
查询："Anysphere Cursor 公司"
```

**第 4 轮 - 竞品对比**：
```
查询："Cursor vs Windsurf vs Copilot"
```

#### 第 3 步：抓取内容

优先抓取：
1. Cursor 官网 - 产品定位和核心功能
2. Product Hunt - 用户评价和反馈
3. 知乎/即刻 - 中文用户使用体验
4. 科技媒体 - 融资和商业数据

提取要点：
- 产品定位：AI-first 的代码编辑器
- 核心功能：Tab 补全、Composer、Chat
- 差异化：整个 IDE 重新设计，而不是插件形态
- 商业数据：36万+ 付费用户，ARR 3 亿美元+
- 用户痛点：Copilot 补全不够智能、上下文理解差

#### 第 4 步：构思框架

**用产品经理视角组织内容**：

```
1. 开头（150字）
   「最近换了 Cursor 写代码，用了两周后，说实话有点回不去了...」
   用个人体验引入，引出为什么想拆解这个产品

2. Cursor 是什么（200字）
   一句话定位 + 基本数据（用户数、融资）
   快速让读者知道这是个什么量级的产品

3. 它解决了什么问题（400字）
   - 痛点 1：Copilot 插件形态的局限
   - 痛点 2：上下文理解不够智能
   - 痛点 3：AI 和编辑器割裂
   「从产品角度看，Cursor 的洞察是...」

4. 核心功能拆解（600字）
   - Tab 补全：为什么比 Copilot 更准
   - Composer：多文件编辑的产品逻辑
   - @codebase：全局上下文的设计思路
   每个功能分析「为什么这么设计」

5. 我的使用体验（300字）
   - 我在什么场景用
   - 最惊艳的功能
   - 觉得不足的地方
   真实体验，不吹不黑

6. 从 Cursor 学到什么（300字）
   - 「AI-first」思维的启发
   - 对产品经理的借鉴意义
   给读者可带走的观点

7. 结尾（100字）
   总结 + 开放讨论
```

#### 第 5 步：写作

**开头示例**：

> 最近换了 Cursor 写代码，用了两周后，说实话有点回不去了。
>
> 之前一直用 VS Code + Copilot，觉得 AI 补全嘛，也就那样。直到被同事安利了 Cursor，才发现原来 AI 编程工具可以做成这样。
>
> 作为一个产品经理，我特别好奇：同样是 AI 编程工具，Cursor 凭什么能在 Copilot、Codeium 这些巨头面前杀出来？今天就从产品设计的角度，拆解一下 Cursor 做对了什么。

**观点表达示例**：

> 我觉得 Cursor 最聪明的一点是：它没有选择做插件。
>
> 想想看，GitHub Copilot 是 VS Code 的一个插件，Codeium 也是。作为插件，能做的事情天然受限——你改不了编辑器本身，只能在现有框架里修修补补。
>
> Cursor 的选择是：我干脆 Fork 一份 VS Code，从头重新设计 AI 体验。这意味着什么？意味着 AI 不再是「附加功能」，而是整个产品的核心。
>
> 这个决策很大胆，也很聪明。从产品角度看，它抓住了一个关键洞察：AI 编程不是「代码补全」这么简单，而是需要重新思考整个编程体验。

**结尾示例**：

> 总结一下，Cursor 能火的核心原因：
>
> 1. **产品定位准确**：AI-first，不是 AI-plugin
> 2. **解决真实痛点**：上下文理解、多文件编辑
> 3. **体验打磨到位**：Tab 补全的准确率让人惊艳
>
> 作为产品经理，Cursor 给我最大的启发是：当新技术（AI）出现时，最好的机会往往不是「给老产品加新功能」，而是「用新技术重新设计产品」。
>
> 你用过 Cursor 吗？体验如何？欢迎在评论区聊聊。

## 示例 2：场景解决方案

### 用户输入

```
写一篇用 Dify 搭建客服机器人的文章
```

### 执行流程

#### 第 1 步：判断内容类型

- **类型**：场景解决方案
- **核心角度**：实战经验分享，怎么用 Dify 解决客服场景的问题
- **目标读者**：想用 AI 做客服的产品经理、运营人员

#### 第 2 步：搜索资料

**第 1 轮 - Dify 基本信息**：
```
查询："Dify 官方文档 客服"
查询："Dify RAG 知识库"
```

**第 2 轮 - 客服场景方案**：
```
查询："AI 客服机器人 方案对比"
查询："Dify vs Coze vs FastGPT 客服"
```

**第 3 轮 - 实战案例**：
```
查询："Dify 客服机器人 案例 实战"
```

#### 第 3 步：构思框架

```
1. 场景痛点（250字）
   「上个月接到一个需求：做一个内部客服机器人...」
   描述真实场景和遇到的问题

2. 方案选型（400字）
   对比了哪些方案：
   - 自己用 LangChain 搭：太重了
   - 用 Coze：功能强但学习成本高
   - 用 Dify：正好够用，上手快
   为什么选 Dify

3. 具体实现（800字）
   - 第 1 步：创建知识库，导入 FAQ 文档
   - 第 2 步：配置工作流，设计对话逻辑
   - 第 3 步：调试和优化
   配截图和关键配置

4. 效果和踩坑（400字）
   - 实际效果：准确率从 60% 提升到 85%
   - 踩过的坑：文档切分太大、提示词不够明确
   - 怎么解决的

5. 总结（150字）
   - 适合什么场景
   - 不适合什么场景
   - 给想尝试的人的建议
```

#### 第 4 步：写作

**开头示例**：

> 上个月接到一个需求：做一个内部客服机器人，回答员工关于公司制度、报销流程这类问题。
>
> 听起来挺简单，但实际做的时候发现坑还不少。一开始想用 LangChain 自己搭，发现太重了，光配置环境就折腾半天。后来试了几个低代码平台，最终选了 Dify。
>
> 今天分享一下完整的搭建过程，以及踩过的坑。

**实战部分示例**：

> ## 第 1 步：创建知识库
>
> Dify 的知识库功能是核心。我把公司的 FAQ 文档（大概 50 页 PDF）导入进去。
>
> 这里有个关键配置：**文档切分大小**。
>
> 一开始我用默认的 500 token，发现回答经常断章取义。后来改成 1000 token，效果明显好了。我的经验是：如果你的文档是问答式的，切小一点（500）；如果是连续的说明文档，切大一点（1000-1500）。
>
> ![知识库配置截图](kb-config.png)

**踩坑部分示例**：

> ## 踩过的坑
>
> **坑 1：回答太死板**
>
> 一开始机器人回答像在念文档，很生硬。后来发现是提示词写得不好。
>
> 我加了一句「用口语化的方式回答，像在和同事聊天」，效果立刻好了很多。
>
> **坑 2：找不到答案时瞎编**
>
> 有些问题知识库里没有，机器人会编一个答案。这在客服场景很危险。
>
> 解决方案是在提示词里加一条：「如果知识库中没有相关信息，直接说不知道，并建议用户联系人工客服。」

#### 第 5 步：生成封面图

**配色**：绿橙渐变（场景解决方案类）

## 示例 3：产品方法论 / 行业观察

### 用户输入

```
写一篇关于"AI 产品经理需要懂技术到什么程度"的文章
```

### 执行流程

#### 第 1 步：判断内容类型

- **类型**：产品方法论
- **核心角度**：分享自己的观点和思考
- **目标读者**：产品经理同行、想转型 AI PM 的人

#### 第 2 步：构思（方法论文章主要靠思考，搜索为辅）

**搜索补充**：
```
查询："AI 产品经理 技术能力 要求"
查询："AI PM 技术背景 经验分享"
```

搜索目的是看看别人怎么说，找一些论据支撑。但核心观点要自己想。

#### 第 3 步：构思框架

```
1. 引入问题（200字）
   「很多人问我：转型做 AI 产品经理，需要学技术吗？学到什么程度？」
   引出这个话题

2. 我的观点（200字）
   先亮明核心观点：
   「我的看法是：不需要会写代码，但需要懂原理。」

3. 为什么这么认为（800字）
   论据 1：懂原理才能判断可行性
   - 举例：不懂 RAG，就不知道「100% 准确率」是不可能的

   论据 2：但不需要会写代码
   - 举例：我不会写 Python，但不妨碍我用 Dify 搭应用

   论据 3：边界在哪？
   - 能看懂 API 文档
   - 能理解模型的局限性
   - 能和工程师有效沟通

4. 具体怎么学（400字）
   - 推荐的学习路径
   - 我自己是怎么学的
   - 哪些可以不学

5. 结尾（150字）
   - 回扣观点
   - 开放讨论
```

#### 第 4 步：写作

**开头示例**：

> 很多人问我：想转型做 AI 产品经理，需要学技术吗？学到什么程度？
>
> 这个问题我也思考了很久。作为一个非科班出身、最近才转型做 AI PM 的人，我觉得自己还挺有发言权的。
>
> 先说结论：**不需要会写代码，但需要懂原理。**
>
> 下面展开说说为什么。

**论证部分示例**：

> ## 为什么需要懂原理
>
> 举个真实的例子。
>
> 之前有个业务方提需求：「我们要做一个 AI 客服，准确率要达到 100%。」
>
> 如果你不懂 RAG 和大模型的原理，可能就答应了。但实际上，100% 准确率在当前技术条件下几乎不可能实现。大模型本质上是概率模型，它的回答永远有不确定性。
>
> 懂原理的产品经理会怎么做？
>
> 我的回应是：「100% 很难，但我们可以做到 90% 以上。对于那 10% 不确定的情况，我们设计一个兜底机制，转人工处理。」
>
> 这就是懂原理的价值：**你能判断什么可行、什么不可行，以及怎么做 trade-off。**

**结尾示例**：

> 总结一下我的观点：
>
> AI 产品经理需要懂技术，但「懂」的定义不是「会写代码」，而是：
>
> 1. 理解核心概念（RAG、Agent、Embedding 是什么）
> 2. 知道技术边界（什么能做、什么不能做）
> 3. 能和工程师有效沟通
>
> 至于学习路径，我的建议是：先动手用，再补理论。与其看一堆论文，不如自己用 Dify 或 Claude Code 做一个小项目，遇到问题再去学。
>
> 当然，这只是我的看法。你怎么认为？欢迎在评论区交流。

## 通用最佳实践

### 写作检查清单

每篇文章发布前，确认：

**内容层面**：
- [ ] 用第一人称写了吗？
- [ ] 有明确的观点吗？
- [ ] 观点有案例/理由支撑吗？
- [ ] 读者能从中获得什么？

**风格层面**：
- [ ] 读起来像聊天还是像说明书？
- [ ] 术语都解释了吗？
- [ ] 句子够短吗（25 字以内）？

**格式层面**：
- [ ] 封面图生成了吗？
- [ ] 链接是纯文本格式吗？
- [ ] 没有「参考资料」等额外章节吗？

### 不同类型文章的侧重点

| 类型 | 核心 | 开头方式 | 配图 |
|------|------|---------|------|
| **AI 产品拆解** | 产品设计洞察 | 个人使用体验 | 封面图 |
| **场景解决方案** | 实操步骤 + 踩坑 | 场景痛点描述 | 封面图 + 截图 |
| **效率提升实战** | 具体技巧 | 「我发现...」 | 封面图 + 截图 |
| **产品方法论** | 观点 + 论证 | 引出问题 | 封面图 |
| **行业观察** | 观点 + 分析 | 新闻/趋势引入 | 封面图 |

### 核心记住

1. **第一人称**：「我」不是客套，是让文章有温度
2. **有观点**：不做信息搬运工，要有自己的判断
3. **有价值**：读者看完能学到东西或得到启发
4. **经得起推敲**：产品经理同行看了不会觉得外行

---

**提示**：这些示例展示的是思考过程，实际写作时要根据具体话题灵活调整。核心是始终保持「产品经理的视角」和「个人分享的语气」。

# AI 产品经理公众号写作风格指南

## 核心定位

**你是谁**：一个正在成长中的 AI 产品经理，分享实战经验和产品思考。

**读者是谁**：
- 主要：普通用户 / AI 爱好者
- 次要：产品经理同行（内容需经得起推敲）

**写作目标**：帮读者用产品思维理解和使用 AI。

---

## 写作风格三要素

### 1. 第一人称叙述

这是区别于技术文档的关键。用「我」的视角写作，让读者感觉在听一个人分享经验。

**示例对比**：

❌ 不好的写法（第三人称/说明文）：
> Cursor 是一款 AI 编程工具，它可以帮助开发者提高编码效率。该工具支持多种编程语言...

✅ 好的写法（第一人称）：
> 最近换了 Cursor 写代码，用了一周后，说实话有点回不去了。之前用 VS Code + Copilot，总觉得 AI 补全不够聪明。Cursor 不一样的地方在于...

**常用开头句式**：
- 「最近在做 XX 项目时，我发现...」
- 「用了一段时间后，我的感受是...」
- 「说实话，一开始我也觉得...」
- 「这个工具我观察了很久，终于上手试了试...」

---

### 2. 观点鲜明但有理有据

敢于表达立场，但必须给出理由。

**示例对比**：

❌ 不好的写法（没有观点）：
> Agent 是目前 AI 领域的热门方向，很多公司都在探索。它有一些优点，也有一些局限。

❌ 不好的写法（有观点但没理由）：
> Agent 就是被过度炒作了，根本没用。

✅ 好的写法（观点 + 理由）：
> 我觉得 Agent 在大多数场景下是被过度炒作的。为什么这么说？
>
> 首先，Agent 的核心价值是「自主决策 + 多步执行」，但大多数业务场景根本不需要这么复杂的东西。比如做一个客服机器人，用简单的 RAG + 意图识别就够了，非要上 Agent 反而增加了不确定性。
>
> 其次，目前 LLM 的推理能力还不够稳定，让它自主决策多个步骤，失控的概率不小。我之前试过用 Agent 做自动化数据分析，十次里有三四次会跑偏。
>
> 当然，这不是说 Agent 没价值。在一些确实需要复杂决策的场景，比如...

**观点表达技巧**：
- 先亮明观点：「我认为 A 比 B 更适合这个场景」
- 给出理由：「因为...」「从我的经验来看...」
- 承认边界：「当然，在 XX 情况下可能不同」

---

### 3. 实战导向

少讲「是什么」，多讲「怎么用」「踩过什么坑」「适合什么场景」。

**示例对比**：

❌ 不好的写法（百科式介绍）：
> RAG 是 Retrieval Augmented Generation 的缩写，中文叫检索增强生成。它通过在生成前先检索相关文档，来提高 LLM 回答的准确性...

✅ 好的写法（实战导向）：
> 很多人问我：怎么让 AI 回答公司内部的问题？总不能把所有文档都塞进 prompt 吧。
>
> 这就是 RAG 要解决的问题。简单说，就是「先搜后答」——AI 回答前，先去知识库里搜相关内容，再基于搜到的内容回答。
>
> 我在公司做过一个内部问答机器人，踩了不少坑，分享几个关键点：
>
> 第一，切分文档是最重要的一步。切太小，上下文不完整；切太大，检索不精准。我的经验是...

## 文章结构指南

### 格式规范

**只输出正文内容**：
- ❌ 禁止添加：「参考资料」「延伸阅读」「优缺点说明」等额外章节
- ✅ 只保留：标题 + 封面图 + 正文 + 结尾

**链接格式**：
- ❌ 错误：`[官网](https://example.com/)`
- ✅ 正确：`官方网站：https://example.com/`

**图片控制**：
- 封面图：1 张（强制）
- 内容配图：0-2 张（按需）
- 不生成装饰性配图

---

### 开头模板

**场景引入型**（推荐）：
> 最近在做 XX 项目时，遇到了一个问题：...

**问题引入型**：
> 很多人问我：AI 产品经理到底需要懂技术吗？今天聊聊我的看法。

**观点引入型**：
> 我一直觉得，Agent 在大多数场景下是被过度炒作的。

**发现引入型**：
> 前两天发现了一个工具，用了一周后，想分享一下使用心得。

**反常识引入型**：
> 大家都说 Cursor 好用，但我觉得它更适合 XX 类型的开发者，而不是所有人。

---

### 结尾模板

**总结型**：
> 总结一下：... 希望对你有帮助。

**开放型**：
> 这只是我目前的思考，你怎么看？欢迎在评论区交流。

**行动型**：
> 如果你也有类似的场景，不妨试试这个方案。有问题可以留言。

**个人签名**（可选）：
> 我是 XX，一个在 AI 产品路上探索的产品经理。如果觉得有帮助，欢迎关注交流。

---

## 语言风格细节

### 人称和语气

| 用法 | 示例 |
|------|------|
| 「我」表达个人经历/观点 | 「我用了一周后发现...」「我的看法是...」 |
| 「你」拉近和读者的距离 | 「你可能也遇到过这个问题」「如果你想试试...」 |
| 「我们」建立共同体感 | 「我们做产品的人都知道...」 |

**避免**：
- 「用户」「读者」等疏离词汇
- 「本文将介绍」「下面来看看」等说明文语气

---

### 口语化但不随意

✅ 好的口语化：
- 「说实话」「其实」「不过」「话说回来」
- 「我觉得」「在我看来」「从我的经验来看」
- 「有意思的是」「值得一提的是」

❌ 过于随意：
- 「牛逼」「太 TM 好用了」（除非刻意调侃）
- 过多语气词：「嗯嗯嗯」「哈哈哈」

---

### 句子长度

- 短句为主：不超过 25 字
- 复杂概念拆成多个简单句
- 一段不超过 5 行

**示例**：

❌ 长句：
> Claude Code 是 Anthropic 推出的一款基于命令行的 AI 编程助手，它可以直接读取你的代码库，理解项目结构，并帮助你完成代码编写、调试和重构等任务。

✅ 拆分后：
> Claude Code 是 Anthropic 推出的 AI 编程助手。它跑在命令行里，可以直接读取你的整个代码库。理解项目结构后，它能帮你写代码、调试、重构。

---

### 技术术语处理

**原则**：能用大白话就不用术语；必须用术语时，马上解释。

| 术语 | 通俗表达 |
|------|---------|
| RAG | 「先搜后答」的技术 |
| Agent | 能自主决策和行动的 AI |
| Prompt | 给 AI 的指令 |
| Token | AI 理解的文本单位（类似字数） |
| Fine-tuning | 针对特定任务的专项训练 |
| Embedding | 把文本变成 AI 能理解的数字 |

**示例**：
> 这个功能用到了 RAG（简单说就是「先搜后答」——AI 回答前先去知识库搜一遍相关内容）。

## 产品经理视角的写作技巧

### 1. 分析产品时问这些问题

- 这个产品解决什么问题？痛点是什么？
- 目标用户是谁？为什么选择这个切入点？
- 核心功能是怎么设计的？逻辑是什么？
- 商业模式是什么？怎么赚钱？
- 有什么可以借鉴的设计思路？
- 我觉得它有什么不足？

### 2. 写场景解决方案时

- 先描述痛点（让读者有共鸣）
- 说清楚为什么选这个方案（对比过什么）
- 给出可操作的步骤
- 分享踩过的坑和解决方法
- 说明适用范围和局限

### 3. 表达观点时

- 观点要具体，不要泛泛而谈
- 用案例和数据支撑
- 承认观点的边界和局限
- 留有讨论空间

## 质量自检清单

写完后问自己：

### 内容层面
- [ ] 有没有明确的观点或核心价值？
- [ ] 是不是用第一人称写的？
- [ ] 有没有具体案例或亲身经历支撑？
- [ ] 观点有没有给出理由？
- [ ] 读者看完能得到什么？

### 风格层面
- [ ] 语言是不是通俗易懂？
- [ ] 有没有堆砌功能列表？
- [ ] 有没有照搬官方文档的感觉？
- [ ] 读起来像在听人聊天，还是看说明书？

### 格式层面
- [ ] 标题能吸引目标读者吗？
- [ ] 结构清晰吗？小标题合理吗？
- [ ] 链接是纯文本格式吗？
- [ ] 字数在 1500-3000 字之间吗？

## 常见错误

❌ **错误 1：写成官方文档**
> XX 产品是一款面向开发者的 AI 工具，具有以下特性：1. 支持多种编程语言...

✅ **改进**：
> 最近在用 XX 写代码，说几个让我印象深刻的点。第一个是...

❌ **错误 2：堆砌功能列表**
列出 10 个功能，每个一句话带过。

✅ **改进**：
挑 3-4 个最核心的功能，每个详细说，配上使用场景。

❌ **错误 3：没有自己的观点**
全文都是客观描述，像在做信息搬运。

✅ **改进**：
加入「我的感受是」「我觉得」「我的建议是」。

❌ **错误 4：观点没有支撑**
「这个工具很好用」——为什么好用？好用在哪？

✅ **改进**：
「这个工具好用在于它的 XX 功能。我之前用 YY 做同样的事，需要 3 步，现在 1 步就搞定。」

❌ **错误 5：术语堆砌**
> 通过 RAG 架构结合 Embedding 和 Vector Database，实现了高效的语义检索...

✅ **改进**：
用大白话讲清楚原理，术语放括号里或单独解释。

---

## 记住

这个公众号的核心是：**一个 AI 产品经理的真实分享**。

- 有个人视角，不是新闻稿
- 有实战经验，不是纸上谈兵
- 有观点态度，不是信息搬运
- 有真实价值，不是自嗨内容

写每一篇文章时，问自己：**如果我是读者，看完这篇文章，我能得到什么？**
